一、MVC的理解

MVC包括MVC、MVP、MVVM

1.1 侠义MVC

mvc是前后端不分离的model view controller 对应springmvc的

1.1.1 mvc含义

  1. model=pojo 数据模型，核心
  2. view=jsp/html+modelAndView 渲染数据
  3. controller=controller，接收用户请求
    1. 前端控制器 dispatch
    2. 后端控制器 controller

1.1.2 ModelAndView

  1. model 响应数据（map结构）/ javaBean
    1. 请求的最后阶段 框架会把数据加入到request作用域
  2. view jsp地址
    1. 框架对视图执行forward重定向操作

1.1.3 model 返回的数据

  1. modelAndView.addObject(“name”,”王勇敢”);其实和request.serAttribute(“name”,”王勇敢”)
  2. 四种数据
    1. obj
      1. json的转换 和 返回设置格式 context:json是固定重复代码
      2. 除去重复代码：基于消息转换器.jackon
        1. 加入依赖 jackson-core jackson-databind 注意版本
        2. 加入注解驱动 注意是mvc的driven
        3. 加@ResponseBody
      3. 返回json原理
        1. 注解驱动的接口httpMessageConverter定义很多obj转换方法，如xml,json,txt
        2. spring注入这个接口的是是实现类。
    2. string(视图路径)
    3. string,int 数据
    4. void使用response.WriteStream响应（只有数据，没有视图）

1.2 MVVM

MVVM是流行的前后端分离model view modelAndView，这个更多的体现在前端

  1. model=controller+pojo
  2. view=html(model)
  3. modelAndView=model-view转换器

1.3 mvc优缺点

1.3.1 优点

  1. 耦合度低：表改了，只需修改model
  2. 复用性高：model可以在不同的视图中使用
  3. 可维护性高
  4. 分层，结构清晰，开发快

1.3.2 缺点：

  1. 适合大型项目，但是现在的小型项目也在使用这种模式，就有点得不偿失
  2. 增加了系统的复杂性，降低了运行效率

Bean声明周期

  1. 整体步骤-摘自
    1. Spring启动，查找并加载需要被Spring管理的bean，进行Bean的实例化
    2. Bean实例化后对将Bean的引入和值注入到Bean的属性中
    3. 如果Bean实现了BeanNameAware接口的话，Spring将Bean的Id传递给setBeanName()方法
    4. 如果Bean实现了BeanFactoryAware接口的话，Spring将调用setBeanFactory()方法，将BeanFactory容器实例传入
    5. 如果Bean实现了ApplicationContextAware接口的话，Spring将调用Bean的setApplicationContext()方法，将bean所在应用上下文引用传入进来。
    6. 如果Bean实现了BeanPostProcessor接口，Spring就将调用他们的postProcessBeforeInitialization()方法。
    7. 如果Bean 实现了InitializingBean接口，Spring将调用他们的afterPropertiesSet()方法。类似的，如果bean使用init-method声明了初始化方法，该方法也会被调用
    8. 如果Bean 实现了BeanPostProcessor接口，Spring就将调用他们的postProcessAfterInitialization()方法。
    9. 此时，Bean已经准备就绪，可以被应用程序使用了。他们将一直驻留在应用上下文中，直到应用上下文被销毁。
    10. 如果bean实现了DisposableBean接口，Spring将调用它的destory()接口方法，同样，如果bean使用了destory-method 声明销毁方法，该方法也会被调用。
  2. 创建声明周期

IOC概念

原理

  1. 解决的问题

    1. 原来一个对象使用另一个对象，只能new

      • 只能new,加入我改变了new 的参数,那我要在很多代码里面改那我要在很多代码里面改，耦合度太高 ，所以创建一个工厂统一管理对象的创建，用到的地方不需要知道怎么new，只需要过去就行了
      • 每次都New 很浪费资源，所以改为通过xml解析获得
      • Class.forName获取,进一步获取bean，进一步降低耦合度

    2. 实际解决问题

      1. 注入bean，，setter等方式

      2. @bean放到方法上，并且这个方法有参数，那么spring就能帮我们从ioc中找到这个参数

        如果是我们自己创建的，ioc根本无法不知道有哪些类被创建了

1.2 @Configruation

  1. 获取Context,这种方法相当于另创一个@Configruation，里面的@Bean需要重新创建一遍

  2. 从Spring3.0，@Configuration用于定义配置类，可替换xml配置文件，被注解的类内部包含有一个或多个被@Bean注解的方法，这些方法将会被AnnotationConfigApplicationContext或AnnotationConfigWebApplicationContext类进行扫描，并用于构建bean定义，初始化Spring容器。

    1. @Configuration不可以是final类型；
    2. @Configuration不可以是匿名类；
    3. 嵌套的configuration必须是静态类

  3. 一个@Configuration下面可以定义多个@Bean

  4. 基于java而没注解@Configuration的注册方法

1.3 @Service,controller,repository和@Component区别

  1. 前三者含有特定功能的Component 比如
    1. dispatchServlet寻找的是@Conroller的GetMapping而不是@Component的mapping
  2. 后者可用前三者代替

1.4 @Mapper和@Repository的区别

  1. 相同点
    @Mapper和@Repository都是作用在dao层接口，使得其生成代理对象bean，交给spring 容器管理
    对于mybatis来说，都可以不用写mapper.xml文件
  2. 不同点
    @Mapper不需要配置扫描地址，通过xml里面的namespace里面的接口地址，生成了Bean后注入到Service层中。@MapperScan(“mapper文件所在包”)，这样就不需要每个mapper文件都加@Mapper注解了
    而@Repository不可以单独使用,需要配置@MapperScan(“com.anson.dao”)

1.5 @Bean和@Component区别

  1. 前者作用于methon，后者作用于class

  2. @Component通过@ComponentScan扫描得到

  3. @Bean在@Configuration Class下

  4. @Bean更灵活—>配置文件调控的时候就只能用@Bean

1.6 @Autowire和@Resource的区别

  1. 作用域相同
    1. 属性
    2. setter
  2. @Autowire 默认先按类型装配
  3. @Resource 默认先按名称装配
  4. @Resource =@Autowrie+@Qualifier
