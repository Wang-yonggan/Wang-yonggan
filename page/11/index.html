一、介绍

1.1 跨域原因

  • 浏览器的同源策略限制：浏览器核心、基本的安全功能，如果缺少了同源策略，则浏览器的正常功能可能都会受到影响。
  • 同源策略：会阻止一个域的js和另外一个域的内容进行交互。http-host-port。
  • 前后端分离：前后端的域不一致。

1.2 没有同源策略会引发的问题

​ 一个网站用户Bob可能正在浏览聊天论坛，而同时另一个用户Alice也在此论坛中，并且后者刚刚发布了一个具有Bob银行链接的图片消息。设想一下，Alice编写了一个在Bob的银行站点上进行取款的form提交的链接，并将此链接作为图片src。如果Bob的银行在cookie中保存他的授权信息，并且此cookie没有过期，那么当Bob的浏览器尝试装载图片时将提交这个取款form和他的cookie，这样在没经Bob同意的情况下便授权了这次事务。

  1. 病毒网站对cookie、session的盗用
  2. 盗版网站对后台资源的复用

1.3 问题

  1. 前端后端的代码需要部署到同一台服务器问题的跨域
  2. 前后端不同服务器

一、IO概述

  1. 同步阻塞IO-面向流-BIO
    1. 一个请求连接一个线程，
    2. 问题：需要创建很多的线程，所有的线程还是同步的
    3. 解决：线程池，但是线程池数量有上限
    4. 适合请求少的场景，程序简单
  2. 同步非阻塞IO-多路复用器-NIO
    1. 一个线程可以处理多个请求
    2. selector 选择器(复用器)
    3. 适合请求多，请求时间短：聊天，弹幕
  3. 异步非阻塞IO–AIO(NIO2.0)
    1. 一个请求一个线程
    2. 适合请求较多，且请求时间长场景

每个请求创建一个线程去处理请求

2.2.3 缺陷

  1. 每个请求都创建线程，线程的切换、竞争会浪费资源
  2. 线程本身也占用cpu资源
  3. 并不是每个请求线程

2.3 伪异步

  • 优：创建线程池，能避免创建线程的资源消耗问题
  • 缺：底层还是阻塞的，如果请求处理较慢，那么新的请求就会长时间的再请求队列中，产生大量超时
  • 适应：小批量，请求处理快速的场景

三、NIO

3.1 NIO概述

3.1.1 基本概念

  1. since JDK1.4
  2. 基于通道的非阻塞IO，IO复用
  3. 支持面向缓冲区

3.1.2 三大核心组件

  1. Channel 通道
  2. Buffer 缓冲区
  3. Selector 选择器(多路复用器)
    1. 轮询每个通道

3.1.3 非阻塞模式

一个线程可以同时处理很多请求，基于监听器

3.1.4 NIO模型图

3.1.5 NIO vs BIO

BIO NIO
数据流(字节，字符) 数据块(缓冲区，通道)
流：单向 通道：双向
阻塞 非阻塞
一个请求需要一个线程 一个线程处理处理多个请求(选择器监听)

3.2 Buffer

3.2.2 Buffer属性

  1. 容量：数组大小,不可更改
    1. 通过allocate分配
  2. 限制：可操作的最后位置
    1. 读：表示有数据的最后位置
    2. 写：表示容量
  3. 位置：写入、读取的当前位置索引
    1. 标记：mark该位置
    2. 重置：重置到mark位置

3.2.3 Buffer Api

  1. clear 清空缓存区
  2. flip：转换成可读模式
  3. Remaining：获得缓冲区长度
  4. hasRemaining：是否有数据
  5. rewind：将位置置0清除mark

3.2.4 缓冲区分类

3.2.4.1 直接缓冲区

写入到直接内存，高并发场景

3.2.4.2 非直接缓冲区

写入堆内存

3.3 Channel

3.3.1 特点

  1. 双向的
  2. 不能直接数据交互，需借助buffer
  3. 异步读写

3.3.2 Channel实现

  1. FileChannel 文件
  2. DatagramChannel UDP
  3. SocketChannel TCP
  4. ServerSockeChannel 监听Web的TCP连接，每一个连接创建一个Channel

3.4 Selector

3.4.1 工作方式

多个 Channel 以事件的方式可以注册到同一个Selector，如果有事件发生，便获取事件然后针对每个事件进行相应的处理。这样就可以只用一个单线程去管理多个通道，也就是管理多个连接和请求。避免了多线程之间的上下文切换导致的开销
